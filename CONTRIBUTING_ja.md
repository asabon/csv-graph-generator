## 開発ワークフロー

このプロジェクトでは、ネイティブコンパイルを必要とする依存関係（`canvas` など）を扱うため、特殊なビルドワークフローを採用しています。

### 重要: ローカルでビルドしないでください

`canvas` の依存関係は一部の環境（特に Windows）でのセットアップが難しいため、**ローカルでのビルドは必須ではなく、推奨もしていません。**

代わりに、機能ブランチに変更をプッシュするたびに、プロジェクトを自動的にビルドし、成果物（`dist/` フォルダ）をコミットする CI ワークフローを設定しています。

1.  **ブランチの作成**: 機能や修正のための新しいブランチを作成します（例: `feature/add-pie-chart`）。
2.  **変更の実装**: `src/` 内の TypeScript ソースコードを編集します。
3.  **プッシュ**: 変更をコミットし、GitHub にプッシュします。
    ```bash
    git push origin feature/add-pie-chart
    ```
4.  **自動ビルド**: GitHub Actions ワークフローが自動的に実行され、依存関係のインストール、ビルドを行い、生成された `dist/index.js` をブランチにコミットし直します。
5.  **プル**: さらなる編集を行う前に、変更（ビルド成果物を含む）をローカルに取り込みます。
    ```bash
    git pull origin feature/add-pie-chart
    ```

### リリースプロセス

1.  `package.json` のバージョンを更新します。
2.  GitHub 上で新しいリリースを作成します。
3.  GitHub Marketplace に公開します（有効なリリースの作成時に自動的に行われます）。

## コードスタイル

*   コードのフォーマットには **Prettier** を使用しています。
*   リンティング（静的解析）には **ESLint** を使用しています。
*   コードがこれらの基準に従っていることを確認してください。CI ワークフローでもフォーマットとリントのエラーチェックが行われます。

## 問題の報告

バグを見つけた場合や機能リクエストがある場合は、まず既存の Issue を検索してください。類似の Issue が存在しない場合は、明確な説明を含む新しい Issue を作成してください。
